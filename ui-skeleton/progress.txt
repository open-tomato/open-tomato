# Progress notes for ui-skeleton iteration 1

General findings for future tasks (broadly applicable; not task-specific):

- Sibling shared packages live under `packages/shared/<name>/`, not `packages/<name>/`. From `packages/ui-skeleton/` the relative path is `../shared/<name>`. Use this when rewriting `workspace:^` → `file:` links.
- Package names of the relevant siblings: `@open-tomato/eslint-config` and `@open-tomato/typescript-config`.
- `@open-tomato/eslint-config` already depends on `@open-tomato/typescript-config` via `file:../typescript-config` (precedent for the file-link approach).
- `@open-tomato/eslint-config` exposes subpath exports `./base`, `./next`, `./react` — choose the right one in `eslint.config.mjs`.
- `@open-tomato/typescript-config` exposes subpath exports `./base`, `./react`, `./next` (note the JSON filename is `nextjs.json` but the export key is `./next`).
- `@open-tomato/eslint-config` already lists `eslint-plugin-storybook` as a dep, so Storybook lint integration does not require adding it to ui-skeleton.
- Root `packages/` directory does not have a top-level `package.json` workspaces entry covering `ui-skeleton` this iteration — keep it standalone and use `file:` links.
- Git repository root is `packages/` (not the monorepo root above it). Paths in `git status`/`git diff` are relative to `packages/` (e.g. `ui-skeleton/...`). `cd` above `packages/` is not a git repo.
- `PLAN_TRACKER.md` is the working checklist that gets edited every task — expect it to show as modified in `git status`; this is bookkeeping, not a dirty tree.
- Bun 1.3+ writes a text-JSON `bun.lock` (not the older binary `bun.lockb`). It belongs at the ui-skeleton package root and should be committed.
- `bun install` may warn `Blocked 1 postinstall. Run \`bun pm untrusted\` for details.` — this is informational, not a failure. Bun blocks postinstalls from untrusted packages by default.
- `lucide-react` is at major version 1.x (1.16.0+ at time of writing), not the older 0.x range. If a task or doc references an older 0.x version range, treat it as outdated and pin to ^1.x.
- `bun add <pkg>` updates `package.json` automatically and reorders keys alphabetically — expect dependency blocks to be sorted after each add.
- Radix UI primitive packages are mostly at major v1.x, but a few are at higher majors (e.g. `@radix-ui/react-label` is v2.x). Don't assume a uniform major across the `@radix-ui/*` namespace when pinning.
- `bun add` of multiple packages in a single command works and is faster than per-package adds; the lockfile is updated once at the end.
- Tailwind v4 ships as `tailwindcss@^4` (resolves to 4.3.x) with companion packages `@tailwindcss/postcss@^4` (PostCSS plugin) and `@tailwindcss/vite@^4` (Vite plugin). The Vite plugin is the preferred integration for Vite-driven builds; the PostCSS plugin covers tooling that runs PostCSS independently (e.g. Storybook in some configs). `postcss` itself is still needed as a dev dep.
- Vite is at major v8 (8.0.13+ at time of writing). Companion plugins for the React + library-mode setup: `@vitejs/plugin-react@^6` and `vite-plugin-dts@^5`. If plan docs or examples reference older majors (Vite 5/6/7), treat the unpinned `bun add` result as authoritative.
- Vitest is at major v4 (4.1.7+ at time of writing) and pairs with `@vitest/coverage-v8@^4` at the same major. `jsdom` is at major v29. `@testing-library/react` is at major v16 (React 19 compatible). `@testing-library/jest-dom` is at major v6. `@testing-library/user-event` is at major v14. `jest-axe` is at major v10 with companion `@types/jest-axe@^3` (the types package lags behind in major version). If older majors appear in plan docs (Vitest 1/2/3, RTL 14/15), treat the unpinned `bun add` result as authoritative.
- Storybook is pinned to `^8` in this iteration (per plan). Storybook 8 packages (`storybook`, `@storybook/react-vite`, `@storybook/addon-essentials`, `@storybook/addon-a11y`, `@storybook/addon-interactions`, `@storybook/test`) resolve to 8.6.x. They declare a Vite peer range that does not include Vite 8, so `bun add` emits `warn: incorrect peer dependency "vite@8.0.13"` — this is a known mismatch, not a blocker for install. If runtime issues surface in Storybook (`storybook dev`/`build-storybook`), expect to either downgrade Vite or upgrade Storybook to a major that supports Vite 8.
- For library packages in this monorepo, react/react-dom belong in `peerDependencies` (broad range like `^19.0.0`) with a dev mirror in `devDependencies` (pinned to the current development version like `^19.2.6`). Consumer apps provide React; the dev mirror keeps local install/test/build working. Manually edit `package.json` (don't use `bun add --peer` plus a separate dep add — easier to keep the two ranges aligned by hand).
- Tailwind v4 stylesheet entry uses a single `@import "tailwindcss";` directive (replaces the v3 trio of `@tailwind base/components/utilities`). Design tokens are declared inside a top-level `@theme { ... }` block as CSS custom properties (e.g. `--color-background`, `--radius`, `--shadow-elev-1`); Tailwind generates matching utilities from them. The package-local token snapshot lives at `src/styles/globals.css`; the sibling `design-system/colors_and_type.css` is referenced manually but not imported this iteration (workspace integration deferred).
- The shared `@open-tomato/typescript-config/base` already sets `moduleResolution: "bundler"`, `target: "ES2022"`, and `lib: ["es2022","DOM","DOM.Iterable"]`. Package-level overrides for these in `tsconfig.json` are redundant but harmless when the values match the base.
- TypeScript `paths` mappings whose values are non-relative (e.g. `"src/*"`) require `baseUrl` to be set, otherwise tsc emits TS5090 ("Non-relative paths are not allowed when 'baseUrl' is not set"). Either add `"baseUrl": "."` or rewrite the values with a leading `./`. The `.` baseUrl resolves to the tsconfig.json directory.
- Running `tsc --noEmit` against an empty package (no `src/` yet) emits TS18003 ("No inputs were found"). This is expected during scaffolding and not a config error — once source files exist it goes away.
- Vite library mode supports a multi-entry build by passing `build.lib.entry` as a `Record<string, string>` (key = chunk name, value = absolute path); combine with `output.entryFileNames: "[name].js"` and `output.preserveModules: false` to emit one file per entry (e.g. `index.js`, `particles.js`, `atoms/Button.js`). Per-atom entries can be generated programmatically via `Object.fromEntries(names.map(...))`.
- `vite-plugin-dts` accepts `tsconfigPath` to point at a separate build tsconfig (`tsconfig.build.json` with `emitDeclarationOnly`) so dts emission is decoupled from the main tsconfig used by editors and `check-types`.
- Externalize peers in Vite library builds via `build.rollupOptions.external` using regex patterns to catch deep imports (e.g. `/^react($|\/)/`, `/^react-dom($|\/)/`, `/^@radix-ui\//`) plus exact strings for non-namespaced deps (`class-variance-authority`, `clsx`, `tailwind-merge`). `lucide-react` is intentionally NOT externalized — it ships as a runtime dep of the library.
- Use `mergeConfig` from `vitest/config` (not `vite`) to layer Vitest options onto the shared `vite.config.ts`; this preserves Vite plugins, the `@` alias, and externals while adding the `test` block (env, setup, coverage). Import the vite config as a default export — Vitest accepts it as an object even though the file exports `defineConfig(...)` from `vite`.
- `bunx vitest run` returns exit 1 with "No test files found" when the package has no `*.test.*` files yet — this is normal during scaffolding, not a config failure. Confirms the config loaded and resolved successfully.
- Avoid `--reporter=basic` for Vitest 4 smoke tests; it tries to resolve `basic` as a custom reporter module and crashes. Use the default reporter (no flag) for config-load checks.
- This package's `lint` script is `eslint --fix` (auto-mutating) rather than the plan example's `eslint .` (check-only). The existing convention is preserved across scaffolding tasks — don't downgrade it to non-fixing unless explicitly asked, because some downstream automation expects `bun lint` to also format.
- `bunx shadcn@latest init --yes` still prompts interactively when a `components.json` already exists (the `--yes` flag does not suppress the overwrite guard). Pipe `N` via stdin to decline; the CLI exits 1 with "To start over, remove the components.json file" — that exit code is "user declined", not a validation failure. The fact the CLI reached the overwrite prompt is itself confirmation that `components.json` parsed cleanly against the shadcn schema; a malformed config errors out earlier with a parse/schema message.
- The ESLint config enforces single quotes in TS/TSX (auto-fixed by `bun lint`). Write source with single quotes from the start to avoid noisy auto-format diffs in commits.
- Pre-existing lint errors live in `README.md` (markdown heading-level/multiple-H1) and `vite.config.ts` (`no-undef` on `__dirname`). They are not caused by particle/atom scaffolding and `bun lint` will continue to exit 1 until those are fixed in their own task. Verify your own file isn't flagged by checking the per-file paths in lint output.
- `pre-prompt.md` (at package root) is the canonical convention doc for per-atom file layout. Every atom under `src/atoms/<Component>/` must have exactly six files: `<Component>.tsx`, `<component>.variants.ts` (lowercase, kebab-case for multi-word), `<Component>.test.tsx`, `<Component>.stories.tsx`, `README.md`, `index.ts`. The Button atom is the canonical reference; new atoms copy its shape.
- ESLint markdown plugin enforces `markdown/fenced-code-language` — every fenced code block in `.md` files needs an explicit language tag. For directory trees / ASCII art use ```text; for shell snippets use ```bash; for code use the appropriate language identifier. The bare ``` opener will fail `bun lint`.
- Atom generation is hand-written (no codegen, no plop/hygen): Button is the canonical reference, every other atom is created by copying Button's six files and adapting per the inventory row in `PLAN.md`. The "Generation workflow" section of `pre-prompt.md` is the authoritative per-atom procedure — read it before scaffolding any new atom. `bunx shadcn@latest add <atom>` may be consulted for the underlying primitive shape, but its output never lands in `src/atoms/` directly; always rewrite into the six-file wrapper convention.
- The ESLint config also enforces import grouping/sorting (auto-fixed by `bun lint`). Expect `import * as React from 'react'` to be reordered alongside third-party imports rather than staying at the top; internal `@/...` imports land in their own group below third-party imports, with `./relative` imports in the last group. Write imports already grouped this way to avoid post-lint diffs.
- When wrapping a Radix primitive that takes a numeric prop (e.g. `AspectRatio`'s `ratio: number`, `Progress`'s `value: number`) with a categorical variant API, export a `<component>Map` const next to the cva call in `<component>.variants.ts` that maps the variant key → numeric value. cva only generates class strings; the numeric mapping is a parallel lookup the component file reads. Pattern likely repeats for any atom whose categorical variant doesn't reduce to a className.
- Radix `AspectRatio` renders two divs: an outer wrapper (gets `data-radix-aspect-ratio-wrapper=""` and inline `padding-bottom` style) and an inner `Primitive.div` (receives the forwarded ref, `className`, and other props). Same shape applies broadly across Radix primitives — props you pass typically target the primary `Primitive.<tag>` slot, not the surrounding wrapper. When testing, find the inner element by the `className`/data-attribute you set, not by document root.
- For atoms where every variant key produces the same visual styling (e.g. `AspectRatio` where the ratio name maps to a numeric prop, not a class), still emit a small marker class per variant in the cva block (e.g. `aspect-[1/1]` / `aspect-[16/9]` / `aspect-[3/4]`) and also set `data-<variant>="<name>"` on the rendered element. The class satisfies the "applies a variant class for at least one non-default variant" test requirement; the data-attribute gives consumers and tests an observable hook.
- Multi-part Radix primitives (e.g. `Avatar` = root + `AvatarImage` + `AvatarFallback`; the same shape applies to upcoming Card/Checkbox-with-label/etc.) fold into the single-entry wrapper convention via slot props rather than sub-component exports. The wrapper renders all parts internally; consumers supply data via flat props (e.g. `src`, `alt`, `fallback`, `fallbackDelayMs`) plus an optional `<part>Props` escape hatch (e.g. `imageProps`) for the inner element. Export multiple cva calls in `<component>.variants.ts` when each part needs its own class block (e.g. `avatarVariants`, `avatarImageVariants`, `avatarFallbackVariants`); the root one drives the public `VariantProps` type, the inner ones are zero-variant style holders.
- When a Radix `*Image` sub-component requires `alt`, default it to `''` in the wrapper rather than making it required at the wrapper API level — callers can omit it for purely decorative avatars/images and still pass a11y checks (axe treats empty alt as "decorative", not "missing"). Make `alt` strongly recommended in the README/JSDoc but not a required TypeScript prop, so the wrapper stays ergonomic for the fallback-only case.
- For polymorphic atoms that use Radix `Slot` (`asChild` pattern) AND also expose icon slot props (e.g. `leadingIcon`/`trailingIcon`), wrap the main `children` in `<Slottable>{children}</Slottable>` and place the icon nodes as raw siblings. `Slot` requires exactly one child to merge with the user's element; `Slottable` (also from `@radix-ui/react-slot`) marks the single mergeable slot and lets other sibling children render alongside without violating that constraint. Under a native element (`asChild=false`) `Slottable` is a transparent Fragment, so the pattern is safe in both branches. Without `Slottable`, `<Slot>` + multiple children throws at runtime when `asChild` is true.
- When wrapping a native `<button>`, default the `type` attribute to `'button'` (allow override via the prop) so the component doesn't accidentally submit ancestor forms. Skip the default when `asChild={true}` because the rendered element may not be a `<button>` (passing `type='button'` onto an `<a>` is invalid). Pattern: `type={asChild ? undefined : (type ?? 'button')}`.
- When an atom exposes a `loading` prop, mirror it onto the DOM via three attributes for the widest accessibility / styling surface: `data-loading=""` (CSS hook + DOM-introspection test hook), `aria-busy="true"` (screen-reader announcement), and `disabled` (block pointer + keyboard activation). Combine with any user-supplied `disabled` so consumers don't need to set both. Tests should assert all three: `toBeDisabled()`, `toHaveAttribute('data-loading', '')`, `toHaveAttribute('aria-busy', 'true')`.
- ESLint's `@stylistic/multiline-ternary` rule will auto-format short inline ternaries across multiple lines on `bun lint` (e.g. `asChild ? Slot : 'button'` becomes a three-line construct). Don't fight the auto-format; expect ternary JSX attribute values like `{cond ? 'a' : undefined}` to be wrapped over three lines after lint. Write them inline and let lint reformat — the resulting diff is cosmetic, not semantic.
- Testing Library's accessible-name match (`getByRole('button', { name: 'X' })`) is exact-match by default; when `leadingIcon`/`trailingIcon` text content is part of the rendered tree, the accessible name becomes `"L X"` (icon + text) and an exact `'X'` lookup fails. Use a regex (`/X/`) or `{ exact: false }` when icons live inside the actionable element. Hidden icons (`aria-hidden`) are still counted by the accName algorithm if they contain text content — only non-text glyphs are skipped.
- When wrapping multi-part shadcn primitives (Card, Accordion, Dialog, etc.) into one slot-based atom, mark each rendered section with `data-slot="<component>-<part>"` (e.g. `card-header`, `card-content`, `card-footer`). The data-attribute lets tests assert section presence/absence (`container.querySelector('[data-slot="card-header"]')`) and resolve which section a piece of content lives in (`screen.getByText('X').closest('[data-slot="card-header"]')`) without coupling to ephemeral class strings, and gives downstream consumers a styling hook that survives className overrides via the `cn`/twMerge collapse.
- For slot-based wrappers (Card-style) where multiple optional slots (`header`, `title`, `description`, `footer`) can independently appear or vanish, gate each section on `prop !== undefined` rather than truthy checks. Truthy checks (`if (title)`) drop legitimate empty-string or `0` slot values; an explicit `undefined` check renders the slot whenever the consumer passes anything at all. Also gate the *children* content slot on `children !== undefined && children !== null` so that a contentless card (header-only or footer-only) doesn't emit an empty inner `<div>`.
- For slot-based wrappers with a "default layout" slot AND an "override" slot (e.g. Card's `title`/`description` vs. a custom `header`), make the override (`header`) replace the entire default layout when present rather than merging — easier mental model for consumers, simpler test surface, and matches established shadcn composition. Document the override semantics in the README's Do/Don't ("DON'T mix the `header` slot with `title` / `description`").
- For atoms that wrap a Radix primitive AND expose an optional inline label slot (Checkbox, Radio, Switch, etc.), auto-generate the linking id via `React.useId()` only when `label !== undefined` and the consumer hasn't passed an `id`. Keep two distinct render paths: when `label` is provided, return an inline-flex `<span data-slot="<component>-root">` containing the Radix trigger + a sibling `<label htmlFor={id} data-slot="<component>-label">`; when omitted, return the bare trigger alone. This keeps unlabelled usage (with `aria-label`) free of extra DOM and matches the accessibility expectation that the label is optional, not always wrapped.
- Native `<label htmlFor>` still provides accessible name for a Radix-rendered `<button role="checkbox">` (and other role-overridden buttons) because the underlying element is a labelable HTML element. `getByRole('checkbox', { name: 'X' })` resolves correctly through the `for`/`id` link — no `aria-labelledby` plumbing required when the wrapper renders its own `<label>`.
- Pair the Radix trigger with class `peer` and put `peer-disabled:opacity-50 peer-disabled:cursor-not-allowed` on the sibling `<label>` so the label tracks the trigger's disabled state via CSS, no extra prop plumbing. Works because Radix forwards native `disabled` onto the underlying `<button>`, so the `:disabled` pseudo-class fires.
- Radix `*Indicator` sub-components (`CheckboxIndicator`, `RadioGroupIndicator`, etc.) only render when the parent's state is non-default (e.g. `checked` or `indeterminate`); they carry the parent's resolved `data-state` so child icons can be discriminated via Tailwind's `group-data-[state=<value>]:` modifier. Pattern: put `group` on the Indicator's className, render both icons (Check + Minus) as children with `group-data-[state=indeterminate]:hidden` / `hidden group-data-[state=indeterminate]:inline-block` — no React state mirroring needed for indeterminate tri-state. The default-named `group` is fine when there is no nested group; use `group/<name>` only if nesting becomes a concern.
- The Radix trigger button itself also gets `data-state="checked" | "unchecked" | "indeterminate"`. Pair `data-[state=checked]:bg-primary data-[state=checked]:text-primary-foreground` (and the same for `indeterminate`) on the trigger's cva block so checked and indeterminate share the filled-in look while unchecked stays default. Saves a `useState` mirror for the visual selected state.
- When a wrapper component's prop name collides with a native DOM attribute (e.g. Card's `title` prop conflicts with HTMLAttributes' `title: string`), use `Omit<React.HTMLAttributes<...>, 'title'>` in the props interface to shadow the DOM attribute with the richer `ReactNode` type. Pattern recurs for any atom that takes ReactNode slot props with names like `title`, `value`, `default` that clash with HTML.
- When an atom has two variant axes that visually couple (e.g. Card's `padding` controls both root `gap-*` AND inner section `p-*`), keep the cva block focused on the axis you can express as a single class on the root, and expose the parallel lookup as a separate `<component><Axis>Map` const (e.g. `cardSectionPaddingMap`). The component file reads the lookup to apply class strings to inner elements while cva still drives `VariantProps` typing. Same pattern as `aspectRatioMap` for numeric variant payloads — generalize whenever a variant key needs to project into more than one class string.
- For wrapper-frame atoms (Input pattern: wrap a native control in a styled `<div>` that carries the visible frame so icon slots can sit alongside the control), use `focus-within:ring-*` on the wrapper instead of `focus:ring-*` on the control, and propagate disabled state to the wrapper visually via `has-[input:disabled]:opacity-50 has-[input:disabled]:cursor-not-allowed`. Always forward `ref` to the inner control element, not the wrapper, so consumer `useRef().current.focus()` and form integration work as expected. `className` becomes ambiguous in this pattern: by package convention apply it to the visible wrapper (the visual escape hatch surface) and forward every other HTML attr to the inner control via `...rest`. Document the split in the README so consumers know where `className` lands.
- Extending `React.InputHTMLAttributes<HTMLInputElement>` (and `TextareaHTMLAttributes`) for variant-prop interfaces requires `Omit<...,'size'>` because the native element exposes a numeric `size` attribute that collides with the `'sm'|'md'|'lg'` union. Same trap will recur for any atom whose categorical variant prop reuses a native-attribute name (`title`, `value`, `default`, `size`).
- For form-control atoms with a semantic `variant: error|success` axis, auto-set `aria-invalid="true"` when `variant === 'error'` so visual error state and a11y error state can never drift. Respect explicit consumer overrides by destructuring the `'aria-invalid'` prop and using `??` (e.g. `ariaInvalid ?? (variant === 'error' ? true : undefined)`); this lets `aria-invalid={false}` deliberately suppress the auto-set value. Same pattern likely applies to `aria-disabled`, `aria-required`, `aria-busy` when a wrapper has a categorical variant or boolean prop that mirrors them.
- `jsx-a11y/label-has-associated-control` is a static-analysis rule and does NOT see through custom components (e.g. `<label><Input/></label>` errors even though `<Input>` does render an `<input>` at runtime). In tests and stories, prefer the `htmlFor`/`id` pairing (`<label htmlFor="x">…</label><Component id="x" />`) over JSX nesting to satisfy the rule without disabling it. The runtime label-association still works either way.
- Tailwind v4's default color palette (red, emerald, slate, etc.) remains available as utility classes even when `@theme` only declares a constrained set of semantic tokens — `@theme` ADDS or OVERRIDES named entries but does not clear the default palette. When a semantic token doesn't exist for a UI state (e.g. no `--color-success` in this iteration), reach for a palette color (`border-emerald-500`) instead of adding a token, and document the choice in the component README so a future iteration can introduce the proper token without breaking the public variant API.
- cva supports boolean variant axes by declaring `variants: { <name>: { true: '<classes>', false: '<classes>' } }` and `defaultVariants: { <name>: false }`. Even when both branches resolve to empty class strings (because the visual effect lives in a separately-rendered sub-element rather than a class on the root), keep the boolean variant in the cva block so it remains part of `VariantProps<typeof xVariants>` and stays discoverable through the public type. Pair it with a corresponding `data-<name>` attribute on the rendered root (set to `''` for true, `undefined` for false) so consumers and tests have an observable hook even when the variant has no direct className impact.
- For decorative visual markers attached to text content (e.g. Label's required-indicator asterisk, info badges, status dots), render the marker as a DOM element with `aria-hidden="true"` rather than a CSS `::after { content: '*' }` pseudo-element. The DOM element approach (i) is reliably ignored by the accessible-name algorithm so `getByRole('textbox', { name: 'Email' })` resolves the linked control's name without picking up the marker, (ii) survives `aria-labelledby` lookups for free, and (iii) gives downstream tests a `data-slot` hook to assert presence/absence. Pseudo-element content is announced inconsistently across screen readers — avoid it for anything semantically meaningful.
- When a wrapper renders an optional decorative element (e.g. Label's required indicator), expose an override prop (`requiredIndicator?: ReactNode`) that defaults to a sensible literal (`'*'`) via the nullish-coalescing operator inside the JSX (`{requiredIndicator ?? '*'}`). Keeps the API ergonomic for the common case while allowing consumers to swap in a richer marker (icon, parenthetical text) without owning the wrapping element or className. Pattern generalizes to any "default literal with optional override" slot.
- For atoms whose underlying primitive type already extends a labelable HTML element interface (e.g. `@radix-ui/react-label` extends `React.LabelHTMLAttributes<HTMLLabelElement>`), no `Omit` is needed when adding variant axes — labels/divs/spans don't carry colliding native attributes like the input element's numeric `size`. Only `<input>`, `<textarea>`, and `<select>` need the `Omit<...,'size'>` dance.
- For Radix track+indicator primitives (`Progress`, and likely `Slider`'s `Range`), drive the indicator width via inline `style={{ transform: 'translateX(-${100-percent}%)' }}` and put `overflow-hidden` on the track's cva block — that's the shadcn-standard pattern and it lets the indicator share a single `w-full` base class regardless of progress. Don't compute the indicator width via `w-[<n>%]` classes; arbitrary-value classes would require Tailwind to generate one utility per render. Pass the raw `value` (including `null`/`undefined`) straight through to the Radix primitive so it can set `data-state="indeterminate"`, but compute the local `percent` after clamping to `[0, max]` so out-of-range inputs don't visually overflow. For `null` values the indicator parks at `translateX(-100%)` (off-screen); we don't render an animated indeterminate state in this iteration.
- When a categorical variant axis needs to drive className subsets on multiple sub-elements of a single wrapper (e.g. Progress's `variant` colors both the track and the inner indicator with different but coordinated palettes), split into two cva blocks (root + `<component>IndicatorVariants` etc.) that each declare ONLY the variant keys they need. The root's `VariantProps` is the public type; the inner blocks are internal styling. Pass the same resolved variant value to both calls in the component file so they stay coordinated. Same shape applies any time a "look" axis projects into more than one rendered element (Toggle pressed-state ring + label, Slider track + range, etc.).
- `toHaveStyle({ transform: 'translateX(-60%)' })` from jest-dom works in jsdom for asserting inline-style props on rendered elements — useful when the visual state of a wrapper (Progress indicator width, Slider range fill, etc.) lives in `style` rather than `className`. Match the exact CSS string Radix/JSX emits; jsdom doesn't normalize `translateX(0%)` vs `translateX(-0%)` so write the expected string in the same form the component produces.
- jsdom does not implement `ResizeObserver`; any Radix primitive that observes its own size (`ScrollArea`, `Slider`, `Toast`, `Tooltip`-positioned content, etc.) will throw `ReferenceError: ResizeObserver is not defined` at render time. Polyfill once in `vitest.setup.ts` with a no-op class (`observe`/`unobserve`/`disconnect` as `vi.fn()`) gated on `typeof globalThis.ResizeObserver === 'undefined'`; the polyfill is broadly safe because the primitives only use the observer for measurement-driven visual state which jsdom can't compute anyway. The shared ESLint config does not include `globalThis` in its globals — add `/* global globalThis */` at the top of `vitest.setup.ts` (or other setup files) to suppress `no-undef` without disabling the rule.
- Radix `ScrollAreaScrollbar` defaults to `type="hover"` which only mounts the scrollbar after the auto-detect logic (`ScrollAreaScrollbarAuto`) measures overflow on its axis. jsdom can't measure layout so only the axis that happens to satisfy Radix's overflow-detection short-circuit will mount — in practice that's vertical only. To exercise both scrollbars in tests, pass `type="always"` so Radix uses `ScrollAreaScrollbarAlways` (unconditional mount). Same trap likely applies to any Radix primitive whose mount logic is gated on layout-derived state (`Popover` collision detection, `Dropdown` positioning, etc.) — when tests need the inner element regardless of layout, look for an "always" / `forceMount` escape hatch on the primitive.
- For multi-part Radix wrappers where a sub-element (e.g. ScrollArea's `Corner`) is conditionally mounted *by Radix itself* based on runtime layout (not by our wrapper's render branch), assert only the wrapper-controlled branches in tests and document the Radix-gated behavior. Don't assert the corner/sub-element's DOM presence; jsdom can't satisfy Radix's gating conditions and you'll get a flaky-looking test. The wrapper's responsibility is "emit the right JSX"; Radix decides whether to mount.
- Multi-part wrappers benefit from exposing per-part escape-hatch props (`viewportProps`, `scrollbarProps`, etc.) typed as `Omit<RadixSubProps, 'children' | 'controlled-by-wrapper-keys'>`. The wrapper merges its own `className` via `cn(componentVariants(...), partProps?.className)` so consumer overrides land on the right inner element. Document in the README which prop targets which DOM node — `className` always means the visible outer frame; inner overrides go via the part props. Same shape generalizes to Card sections, Dialog parts, Accordion items, etc.
- Several Radix primitives already emit `data-orientation="<value>"` on their rendered root automatically (e.g. `Separator`, `ScrollArea` scrollbars, likely `Slider`, `ToggleGroup`, `Tabs`). The wrapper doesn't need to set `data-orientation` itself — assert against the Radix-provided attribute in tests. Still add `data-slot="<component>"` and any extra variant attributes (`data-variant`) the wrapper itself drives, since those aren't provided by Radix.
- Radix primitives that accept a `decorative` boolean (e.g. `Separator`) flip the rendered role between `role="separator"` (semantic, with `aria-orientation` on vertical) and `role="none"` (visual-only, dropped from the a11y tree). Wrappers should default `decorative={true}` to match shadcn's convention but expose the prop so consumers can opt into semantic dividers. Tests use `getByRole('separator')` for the decorative=false branch and `container.querySelector('[data-slot=...]')` for the decorative=true branch (since `role="none"` makes the element unfindable via role queries). Same pattern likely applies anywhere Radix exposes a `decorative` toggle.
- When a wrapper's variant axis shares a name with a Radix primitive prop (e.g. Separator's `orientation` axis collides with Radix's `orientation` prop), use `Omit<RadixProps, '<axis>'>` in the wrapper's props interface even though the value unions look identical. cva's resolved `VariantProps` type widens to `T | null | undefined`, but Radix typically accepts only `T | undefined` — the `Omit` lets the wrapper's resolved-then-passed value satisfy Radix's stricter type after the local `?? 'default'` normalization, with no `as` cast required.
- Tailwind v4 ships `animate-pulse`, `animate-spin`, `animate-ping`, `animate-bounce` as default utilities. Adding a new animation utility requires two CSS additions: a `--animate-<name>: <keyframes-name> <duration> <timing> <iteration>;` token inside `@theme` (which generates the `animate-<name>` utility), AND a sibling top-level `@keyframes <keyframes-name> { ... }` block defining the actual keyframes. `@keyframes` does NOT go inside `@theme` for v4 — it sits at the stylesheet's top level alongside `@theme` and `@layer`. The keyframes name (e.g. `skeleton-wave`) can differ from the utility name (e.g. `animate-wave`); the token's value is what binds them. Pattern recurs whenever a new animation token is needed (shimmer, slide-in, fade, etc.).
- `tailwind-merge` treats `bg-<color>` (background-color) and `bg-gradient-to-<dir>` (background-image) as separate groups — they coexist on the same element rather than the gradient overriding the color. Useful when a wrapper's base class block sets `bg-muted` and a variant adds `bg-gradient-to-r from-* via-* to-*`: both classes survive the merge, the gradient image paints on top of the color, and you don't need to lift the color into every variant branch.
- For "purely decorative placeholder" atoms (Skeleton and likely future loading-state atoms), don't auto-set `role="status"` / `aria-live` on the rendered element — a single placeholder shouldn't be its own live region. Render a plain `<div>` (no ARIA role by default), document in the README that consumers should wrap one or more placeholders in a parent `<div role="status" aria-live="polite" aria-label="Loading">` when announcing loading state. Matches shadcn's convention and keeps the atom composable inside larger loading layouts without producing axe duplicate-role warnings.
- For atoms where the variant axis controls SHAPE but not DIMENSIONS (Skeleton's `rect`/`circle`/`text`, future Divider line-length, etc.), `className` becomes the documented channel for sizing (`w-*`, `h-*`, `size-*`). This is the one case where `className` is NOT the discouraged escape hatch — it's the intended sizing API. Call this out explicitly in the README's "Sizing" or "Props" section so consumers don't confuse it with the color/border `className` overrides that the rest of the package discourages.
- Radix multi-thumb primitives (`Slider`, and likely `RangeSlider`-style future atoms) put `role="slider"` on the inner thumb `<span>`, NOT on the Root. The `aria-label` / `aria-labelledby` a consumer passes to the wrapper lands on the Root by default and is invisible to axe's `aria-input-field-name` rule (which only checks the role-bearing element). The wrapper must destructure `aria-label` / `aria-labelledby` from props and forward them onto every rendered thumb, otherwise `getByRole('slider', { name: '...' })` returns null AND axe flags every thumb. Pattern likely recurs for `Tabs` (role on TabsTrigger), `RadioGroup` (role on RadioItem), `ToggleGroup` (role on Item) — whenever the role is on a sub-element generated by the wrapper, propagate the accessible name there.
- For multi-thumb wrappers, expose an additional `thumbAriaLabels?: string[]` prop so range sliders can give each thumb a semantically distinct name (e.g. `['Min', 'Max']`). Per-thumb override falls back to the root's `aria-label` when an index is missing or the array isn't provided. Same shape applies to any wrapper that emits N role-bearing children from a single set of props.
- For wrappers around Radix primitives whose Root takes an array `value` / `defaultValue` and renders one sub-element per entry (Slider thumbs, ToggleGroup items in `type="multiple"`, etc.), determine the render-count locally in `useMemo` via `Array.isArray(value) ? value : Array.isArray(defaultValue) ? defaultValue : [fallback...]`. Without this the wrapper can't know how many sub-elements to emit; Radix only manages the index-to-state mapping, not the JSX. The fallback for a "no value given" slider is `[min, max]` (two thumbs at the extremes) which makes the range fill visible by default.
- Radix Slider's Root renders an HTMLSpanElement (not div). When typing the `forwardRef`, use `React.forwardRef<HTMLSpanElement, SliderProps>` — same `<span>` ref target applies to Track, Range, and Thumb. Sub-element ref types matter when downstream code reads `ref.current.style` and the consumer expects a span vs. div distinction.
- The standard shadcn-style Slider tonal palette (`bg-secondary` track + `bg-primary` range/border) reads the same Tailwind tokens as Progress (`bg-secondary` track + `bg-primary` indicator). Reuse this pairing for any track+fill atom; it keeps the visual language consistent across Progress, Slider, and any future range/value-display widgets without needing a new semantic token.
- `bun test` is Bun's built-in test runner and does NOT honour the project's Vitest config (no jsdom, no setup file, no `vi` global). Always run package tests through `bun run test` (which executes the package script `vitest run`) or `bunx vitest run`; invoking `bun test <path>` directly errors with `ReferenceError: document is not defined` and `prepareDocument` failures from `@testing-library/user-event`. Same trap applies to any `bun test ...` shorthand — only the script form goes through Vitest.
- For autoResize-style behaviour on a `<textarea>` (height grows to content), the canonical pattern is: `useRef` the element + a `useCallback` `adjustHeight` that sets `el.style.height = 'auto'` then `el.style.height = el.scrollHeight + 'px'`; call it from a `useLayoutEffect` whose deps include `value` and `defaultValue` (catches controlled updates) AND from the `onChange` handler (catches uncontrolled typing). Pair with `resize-none` + `overflow-hidden` classes on the element so manual drag-resize and the inner scrollbar don't fight the JS-driven height. The "fixed `min-h-*` floor from the size variant, no max" rule is documented as the sizing contract; consumers can layer `max-h-*` via `className` if they want a ceiling. Pattern generalizes to any auto-sizing text control (`contenteditable`, code editors, etc.) — the only inputs the technique needs are scrollHeight observability and a place to set inline height.
- `useLayoutEffect` deps for controlled-vs-uncontrolled value tracking should include BOTH `value` and `defaultValue`, not just `value`. Otherwise the initial uncontrolled render's mounted-with-defaultValue state won't trigger an autoResize on mount when `value` is undefined. Same applies to any effect that needs to fire once a control's "current text" is known regardless of mode.
- When a forwarded ref AND an internal ref both need to point at the same DOM node (e.g. autoResize needs to read `scrollHeight`, but the consumer also passed a ref), use a `useCallback` ref combiner: assign to `internalRef.current` first, then call the forwarded ref (function form) or write to `.current` (object form). The combiner's deps are `[forwardedRef]` only — don't put internalRef in deps; it never changes identity. Type the writable assignment as `(forwardedRef as React.MutableRefObject<T | null>).current = node` to silence the readonly-current complaint TypeScript emits on `React.Ref<T>`.
- For testing autoResize behaviour in jsdom, mock the element's `scrollHeight` via `Object.defineProperty(el, 'scrollHeight', { configurable: true, value: <px> })` BEFORE triggering the change/typing. jsdom returns 0 for scrollHeight by default (no layout), so without the mock the test only asserts that `style.height` was set to `'0px'`, which is a weak signal. With the mock, the test verifies the full read-then-write loop: `style.height` lands on the mocked px value.
- For atoms that ARE the loading indicator (Spinner) rather than placeholders (Skeleton), it IS appropriate to auto-set `role="status"` + `aria-label` + an `sr-only` child carrying the label text — Skeleton's "no implicit role" rule (progress note #84) applies to decorative content-shape placeholders, not to active status indicators that stand alone with no other labelled content. Expose a `label?: string` prop defaulting to a sensible value (`'Loading'`), treat `label === ''` as the "decorative" branch (set `aria-hidden="true"`, drop role + aria-label + sr-only, used when the spinner sits next to a separately-labelled message), and respect consumer-supplied `role` / `aria-label` via `??` so explicit overrides always win. Pattern generalizes to any future "active status / progress" atom that can either stand alone or be embedded inside an already-labelled container.
- For ring/border spinners (transparent-top circle with `animate-spin`), drive color through `text-<token>` + `border-current` rather than `border-<color>` directly. cva variant classes touch `text-*` only, the base class block hard-codes `border-current border-t-transparent rounded-full`. This collapses to one base block + one variant axis class regardless of how many colors you add — the variant axis stays at four short class strings instead of four pairs of `border-*` + `text-*`. Same trick works for any visual that wants a single "tonal" color to drive multiple paint-related properties (background, fill, stroke) — pick `text-*` as the canonical channel and reference it via `*-current` everywhere downstream.
- Arbitrary-value Tailwind classes (`border-[3px]`, `w-[7.5rem]`, `aspect-[16/9]`) are fine to use sparingly inside cva variant blocks where the design-token vocabulary doesn't cover the case. Each one generates exactly one utility class at compile time; the cost only matters if the arbitrary value depends on runtime data (in which case it would re-generate every render). For static variant axes the arbitrary value is fixed and gets de-duped, so prefer it over inflating `@theme` with one-off token entries.
- Tailwind v4 ships `sr-only` as a default utility (visually-hidden but exposed to assistive tech), as well as `border-current` and the `border-t-transparent` family. No `@theme` additions needed for accessibility-affordance classes — write them directly in component class strings.
- For polymorphic atoms (Typography-style, where one component renders as one of N possible HTML tags), keep the `as` prop constrained to a closed string union (e.g. `'h1' | 'h2' | ... | 'p' | 'span' | 'code'`) rather than using the generic `as`-prop dance (`<T extends ElementType>`). The constrained union plays nicely with `forwardRef<HTMLElement, …>`, doesn't require complex `ComponentPropsWithoutRef<T>` generics, keeps Storybook controls usable, and matches the design-system intent (consumers shouldn't be rendering Typography as a `<div>` or `<table>`). Inside the component, cast once via `const Component = resolvedAs as React.ElementType;` — the runtime value is a valid intrinsic element string, but `React.ElementType` is the type ReactNode accepts; the cast collapses the union to that contract without losing the ref type.
- For atoms whose visual axis (`variant`) and semantic axis (`as` tag) are decoupled, expose a `variantToTag` lookup const next to the component that maps each `variant` value to a sensible default tag (`display → h1`, `body → p`, `code → code`, etc.). Default `as` falls back through `as ?? variantToTag[resolvedVariant]` so consumers get correct semantics for free, but can override when document outline rules require divergence (e.g. `variant="h2" as="h1"`). Reflect both on the DOM as `data-variant` and `data-as` so downstream code and tests can read the resolved values. Pattern generalizes to any atom where a visual style should *suggest* but not *force* a semantic element.
- When a `variant` axis name collides with another atom's *name* (e.g. Typography's `kbd` variant vs. the dedicated `Kbd` atom that renders `<kbd>`), still ship the variant — but document the divergence explicitly in the README so consumers pick the right tool: the standalone atom for the semantic element, the variant for the visual treatment applied to a different (typically non-semantic) tag. Same pattern likely recurs whenever a visual-style axis shares vocabulary with a semantic primitive (`code`, `mark`, `samp`, etc.).
- For CVA axes that all map to a single utility class (e.g. Typography's `weight: { light: 'font-light', regular: 'font-normal', medium: 'font-medium', semibold: 'font-semibold', bold: 'font-bold' }` and `align: { left: 'text-left', center: 'text-center', right: 'text-right', justify: 'text-justify' }`), inline them in the cva block rather than reaching for particles helpers — the mapping is the contract, and there's no shared logic to extract. Reserve particles for cross-component class strings (focus rings, disabled states) where dedup matters.
- For axes that augment a base variant's styling (Typography's `weight` augments `variant`'s default weight, Toggle's `pressed` augments `variant`'s default tone, etc.), don't set a `defaultVariants` entry for the augmenting axis. Leaving it undefined means cva emits no augment class, so the base variant's default styling holds. `tailwind-merge` would resolve the conflict anyway via `cn()`, but skipping the default keeps the resolved className string shorter and the variant intent more obvious in the DOM.
- `export * from './<layer>'` from a barrel file whose target only contains `export {};` triggers ESLint's `import/export` rule with `No named exports found in module './<layer>'`. The placeholder higher-layer barrels (molecules/organisms/providers/templates/pages) are intentionally empty this iteration but the top-level `src/index.ts` needs to wire them up so future iterations don't have to touch the root barrel. Add a per-line `// eslint-disable-next-line import/export -- placeholder barrel until <layer> ship` comment above each empty re-export rather than refactoring the placeholders to a named-export form; this keeps the wiring stable and removes the suppression cleanly once the layer ships real exports. Pattern recurs anywhere a stable parent barrel needs to reference still-empty child barrels.
- The internal `registry.json` follows a kebab-case `name` convention per item (e.g. `aspect-ratio`, `scroll-area`) that matches shadcn's published-item naming, not the PascalCase atom directory name. The `files[].path` entries are repo-relative (e.g. `src/atoms/AspectRatio/AspectRatio.tsx`) and the variants file is the lowercase/kebab-case form (e.g. `aspect-ratio.variants.ts`). `registryDependencies` in this package's convention lists the *NPM package names* the wrapper imports (e.g. `@radix-ui/react-slot`, `@radix-ui/react-checkbox`, `lucide-react`) — not other registry-item names. Pure-CVA atoms with no external imports beyond `react` + `@/particles/cn` use an empty array. Use `bun -e "const r = await Bun.file('registry.json').json(); console.log(r.items.length)"` as a quick JSON-validity + count sanity check after edits.
- ESM config files (`vite.config.ts`, etc.) don't get `__dirname` / `__filename` from CommonJS; the ESLint config doesn't add them as globals so any reference fails `no-undef`. Use `import.meta.dirname` (Node 20.11+, Bun-supported) for the equivalent path. Bind it once at module top (`const rootDir = import.meta.dirname;`) and reuse — keeps the resolve() calls readable and avoids sprinkling `import.meta.dirname` everywhere. Same approach applies to any new ESM `.config.{ts,js,mjs}` file in this package.
- Markdown lint rules `markdown/no-multiple-h1` and `markdown/heading-increment` enforce a strict outline: exactly one H1 per file (the document title), and no level-skipping between headings. The README's section list uses `### <Category>` (Atoms, Molecules, Organisms, Providers, Templates) — keep new top-level sections at H2 (`## Section`) and category lists at H3 to satisfy both rules. Auto-format does not fix these (the linter can't disambiguate intended structure); fix the heading level by hand.
- The `vite build` output uses two divergent directory layouts: JS files are flattened per `entryFileNames: '[name].js'` (e.g. `dist/atoms/Button.js`, `dist/particles.js`), while `vite-plugin-dts` preserves the source directory structure for declarations (e.g. `dist/atoms/Button/Button.d.ts` + `dist/atoms/Button/index.d.ts`, `dist/particles/index.d.ts`). When the iteration to wire up `package.json` `exports` lands, the `import` path needs to point at the flat JS file and the `types` path needs to point at the nested `index.d.ts` (or `<Name>.d.ts`) — they will NOT share a common stem. Same trap applies to any future entries added to `vite.config.ts`'s `build.lib.entry`.
- Vite's library mode with multiple entries automatically extracts shared modules into hashed chunks at `dist/` root (e.g. `dist/cn--EPVZrAb.js` for the cn helper imported by every atom + index, `dist/Checkbox-<hash>.js` when an atom's content is also pulled in by `dist/index.js`). The per-entry files (`dist/atoms/<Name>.js`) become thin re-exports referencing the shared chunks via relative imports. This is the default Rollup behaviour for `formats: ['es']` multi-entry builds and is acceptable — atom entries still exist as expected, and consumers importing from a deep subpath get the shared chunk transparently. If a flatter dist (one self-contained file per entry) is ever required, set `output.preserveModules: true` or `output.manualChunks: () => null`; both have tradeoffs (more files / larger bundles respectively) so don't change the default without a concrete reason.
