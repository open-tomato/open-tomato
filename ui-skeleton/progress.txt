## Layout conventions
- Path alias `@/...` resolves to `src/...`. Never use deep relative paths like `../../particles/cn`. Sibling imports within an atom/molecule dir use `./<name>`.
- Six files per atom OR molecule (no exceptions): `<Component>.tsx`, `<component>.variants.ts`, `<Component>.test.tsx`, `<Component>.stories.tsx`, `README.md`, `index.ts`.
- Barrel `index.ts` files are two-line `export * from './<X>'` — no side effects, no default exports anywhere under `src/atoms/` or `src/molecules/`.

## Cardinal rules
- Atoms and molecules MUST NOT accept public `className`. The fix is `Omit<..., 'className'>` on the HTMLAttributes-derived props interface AND no consumer string in `cn(...)`. Without `Omit`, `className` is inherited and leaks to DOM via `{...rest}`.
- Per-part bag props that forward to inner sub-components (e.g. Avatar's `imageProps`, ScrollArea's `viewportProps`/`scrollbarProps`) must also include `'className'` in their `Omit<...>` — otherwise consumer strings reach inner parts through the bag and re-open the escape hatch.
- Variants are the only public styling surface. If a knob isn't covered, add a variant axis — don't open an escape hatch.
- Molecules import atoms + particles only; never other molecules or upward layers (organisms/templates/pages/providers). Atoms import particles only. Enforced at lint-time by `no-restricted-imports` blocks in `eslint.config.mjs` keyed off `files: ['src/atoms/**/*.{ts,tsx}']` and `files: ['src/molecules/**/*.{ts,tsx}']`.
- Use `React.forwardRef` with named exports; set `displayName` immediately after.

## Verification chain (always run after a change)
- `bun run check-types && bun run test && bun run lint` — all three must pass.

## Gotchas
- `bun test <path>` is Bun's built-in runner, NOT Vitest. Use `bun run test` or `bunx vitest run` so jsdom + `vitest.setup.ts` (incl. ResizeObserver polyfill) load.
- `bun run lint` is auto-mutating (`eslint --fix`), not check-only.
- Tailwind v4 silently drops utility classes that reference undeclared `@theme` tokens — no build warning. Verify every `bg-*`/`text-*`/`border-*` has a matching `--color-*` in `src/styles/globals.css`.
- `<input>`/`<textarea>`/`<select>` carry a native numeric `size` attribute. Variant interfaces extending `*HTMLAttributes` MUST `Omit<..., 'size'>` to allow categorical `'sm' | 'md' | 'lg'`.
- Multi-thumb Radix primitives (Slider) put `role="slider"` on the THUMB, not the Root. Forward `aria-label` per thumb.
- Radix `Slot` + multiple children throws at runtime when `asChild=true`. Wrap the main child in `<Slottable>` so icon siblings coexist.
- ESM `.config.{ts,js,mjs}` files don't get `__dirname`; use `import.meta.dirname` (Node 20.11+, Bun-supported).
- jsdom does not implement `ResizeObserver`. Any Radix primitive that observes size throws at render without the polyfill in `vitest.setup.ts`.
- The shared `@open-tomato/eslint-config` base ignores `dist/**`, `build/**`, `node_modules/**` but NOT `storybook-static/**`. The package's local `eslint.config.mjs` now spreads the base config and appends `{ ignores: ['storybook-static/**'] }` so `bun run lint` stays clean after `bun run build-storybook`. If you add a sibling package, extend its eslint config the same way (`export default [...baseConfig, { ignores: ['storybook-static/**'] }]`) instead of relying on `--ignore-pattern` flags or deleting build output.
- `no-restricted-imports` matches purely on the import-path string — the target does NOT need to resolve to a real file. To verify a new layer-import guard, drop a temp file under the guarded directory that imports a non-existent path (e.g. `import type { Foo } from '@/molecules/X'`) and run `bun run lint`; the guard fires before any unresolved-import rule complains. Useful when introducing organism/template/page guards before those layers have code.
- Radix Portal content renders OUTSIDE the test's bound `container` (Popover, Tooltip, HoverCard, ContextMenu, Select, and any future portaled organism). `getByRole(...)` against `container` misses portaled content; use `screen.findByRole(...)` (the `screen.find*` family searches the entire `document.body` and awaits async open). Same rule for testing portaled organisms later.
- Radix-trigger wrappers that call `<Radix*.Trigger asChild>{trigger}</Radix*.Trigger>` MUST type the slot as `trigger: React.ReactElement` (NOT `React.ReactNode`). `asChild` uses `React.cloneElement`, which requires a single element child — fragments, strings, arrays, or `null` throw at runtime. Surfacing the constraint at compile time prevents downstream consumers from passing a fragment.

## Markdown / ESLint constraints
- Per-component READMEs: single H1, H2 top-level, H3 nested — no skipped heading levels (`markdown/heading-increment` fails lint).
- Every fenced code block needs a language tag (`text`, `bash`, `tsx`, `ts`, `css`).
- To embed a fenced code block INSIDE another fenced block (e.g. a `markdown` example that itself contains a `ts` snippet), the outer fence MUST use four backticks (` ```` `). Three-backtick outer + three-backtick inner causes the inner closing fence to terminate the outer block early, then the next ``` opens a language-less block and trips `markdown/fenced-code-language`.
- Single quotes in `.ts` / `.tsx`.
- `@stylistic/implicit-arrow-linebreak` forbids a linebreak between `=>` and a wrapped multi-line call. Either keep the body on one line or write `(...) => cn(\n  a,\n  b,\n);` (i.e. start the call on the same line as `=>` and break inside the parens).

## Particle composition pattern
- When an atom needs the shared `wrapperFrameVariants` particle plus its own element-specific selectors (e.g. Input's `has-[input:disabled]:*`), wrap the particle in a function that returns `cn(particle(props), atomSpecificClasses)` and export `type FooVariants = WrapperFrameVariants`. Do NOT re-declare cva for the same axes — duplication drifts.
- To "substitute" a single particle axis branch (e.g. Textarea swapping `h-7` for `min-h-7` in density=compact), pass the neutral value (`density: 'comfortable'`) to the particle so its branch emits nothing, then append the atom-specific replacement string conditionally in `cn(...)`. Avoids tailwind-merge mismatches across `h-*`/`min-h-*` groups and keeps the substitution intentional rather than additive.

## Molecule variant propagation
- Molecules own a lookup table from their public axis to each composed atom's axis (e.g. Alert's `cardPaddingForSize` / `titleVariantForSize` maps `size: 'sm' | 'md' | 'lg'` to Card `padding` and Typography `variant` respectively). Direct passthrough is fine when values match; explicit mapping is required when axes diverge or names differ. Never pass `className` into the composed atom — atoms reject it both at the type level and at runtime.
- Default slot-prop vocabulary across molecules: `leading`, `title`, `description`, `header` (overrides default title/description layout), `actions`, `footer`, `trigger` (Radix-trigger molecules only). Slot content renders raw inside composed atoms — molecules do NOT inject styling into consumer-supplied nodes.
- Molecule tests verify variant propagation by querying the composed atom's data-attributes: e.g. `data-padding="lg"` on the Card root proves the molecule's `size="lg"` mapped to the Card's `padding="lg"` axis; `data-variant="h3"` on the rendered title proves the molecule's size axis mapped to Typography's variant axis. This avoids brittle className introspection and keeps assertions resilient to Tailwind class churn.
- Composition molecules that own N homogeneous atom children (ButtonGroup, ToggleGroup) propagate the group-level axes via `React.Children.map` + `React.isValidElement<Partial<AtomProps>>(child)` + `React.cloneElement(child, { size: child.props.size ?? size, variant: child.props.variant ?? variant })`. The `??` order makes per-child overrides win over the group default — never the other way around. Non-element children (strings, null, fragments) pass through unchanged via the `!isValidElement` early return.
- Segmented-control wrappers (`attached` axis) collapse neighboring borders with neighbor-sibling Tailwind on the wrapper: `[&>*]:rounded-none [&>*:first-child]:rounded-l-md [&>*:last-child]:rounded-r-md [&>*:not(:first-child)]:-ml-px` (and the `-mt-px` vertical equivalent). The wrapper also needs `[&>*:focus-visible]:relative [&>*:focus-visible]:z-10` so the focused child's ring isn't clipped by the next sibling's negative margin. Express these as cva `compoundVariants` keyed on `orientation` + `attached` rather than baking them into each child atom.

## Polymorphic `as` axis (Typography, Item)
- For a polymorphic component whose `as` prop spans elements with divergent attribute surfaces (e.g. Item's `'div' | 'li' | 'button' | 'a'` accepts `href` for `a`, `type` for `button`), extend `React.AllHTMLAttributes<HTMLElement>` instead of `React.HTMLAttributes<HTMLElement>`. `AllHTMLAttributes` is React's union-of-all-HTML-element-attrs type and surfaces `href`, `target`, `rel`, `type`, `disabled`, etc. on the same props interface — no discriminated union needed.
- The `Omit<...>` clause on a polymorphic props interface must include `'title' | 'className' | 'size' | 'color' | 'as'` at minimum. `AllHTMLAttributes` carries `size?: number` (form attribute) and `color?: string`, both of which collide with the categorical `'sm' | 'md' | 'lg'` axes and the semantic `color` variant axes that molecules typically expose. `as` collides with the polymorphic prop we declare ourselves.
- The render call uses `const Component = resolvedAs as React.ElementType; return <Component .../>` — matches Typography's existing pattern. The cost of this simplification is that consumer-supplied attrs are NOT narrowed per-element (a `href` on `as='div'` will type-check but render as an invalid DOM attribute). Document the supported per-element attrs in the README rather than encoding them as a discriminated union.
- When the polymorphic Component renders a native `<button>` (e.g. Item with `as='button'`), inject `type='button'` by default to prevent accidental form submission. Use `const buttonTypeProps = resolvedAs === 'button' ? { type: 'button' as const } : {};` and spread BEFORE `{...rest}` so a consumer can still override with `type='submit'` when the row genuinely lives inside a `<form>`.
- For Typography children inside polymorphic roots: when the root may render as `<button>` or `<a>`, force the composed Typography's `as` to `'span'`. Otherwise Typography's default tag mapping (e.g. `variant='body'` → `<p>`, `variant='h4'` → `<h4>`) produces invalid HTML (`<p>` and `<h*>` cannot appear inside `<button>` per the HTML spec).

## Inline-style atoms (Skeleton, Progress, etc.)
- When an atom owns the inline `style` channel to publish runtime-computed values (Skeleton's width/height/size, Progress's indicator translateX), include `'style'` in the `Omit<HTMLAttrs, ...>` clause alongside `'className'`. Otherwise a consumer-supplied `style` prop reaches the DOM via `{...rest}` and silently races with the atom-emitted style object.
- Dimension props replacing the old `className="h-* w-*"` channel use the shape `width?: string | number; height?: string | number; size?: string | number;` with a `toCss` helper that emits numbers as `${n}px` and passes strings through. `size` overrides both `width` and `height` when provided. Tests assert via `toHaveStyle({ width: '...', height: '...' })`.

## Radix wrapper molecules with auto-injected trigger icons (Collapsible and similar)
- When a Radix-trigger molecule auto-injects an icon (e.g. a rotating chevron) into the trigger's `leadingIcon` / `trailingIcon` slot via `React.cloneElement`, type the slot prop as `React.ReactElement<{ leadingIcon?: React.ReactNode; trailingIcon?: React.ReactNode }>`. The generic on `ReactElement` propagates into `cloneElement`'s constraint so TypeScript flags consumers that pass an element whose props don't accept those slots — catches the "user passed a raw `<div>` as trigger" case at compile time instead of silently no-op'ing the icon injection.
- To drive own decorative `data-state` attributes (e.g. a chevron span using Tailwind's `data-[state=open]:rotate-180` selector) from a Radix wrapper, track open state internally with `useState` and forward `open` + `onOpenChange` as a controlled passthrough to Radix. Radix's `data-state` only lives on the Trigger element itself — siblings/decorative slots can't observe it via CSS without the molecule maintaining a parallel state copy. Pattern: `const isControlled = open !== undefined; const resolvedOpen = isControlled ? open : uncontrolledOpen;` and call the consumer's `onOpenChange` after updating the internal state when uncontrolled.
- When the molecule always passes `open={resolvedOpen}` to a Radix Root, do NOT also pass `defaultOpen` to Radix — `open !== undefined` makes Radix treat the component as controlled and `defaultOpen` becomes a no-op (with a dev warning). Consume `defaultOpen` as the initial value of the internal `useState` and destructure it out of the rest spread so it does not reach Radix.

## Inline-label molecules (Switch, Checkbox-atom, RadioGroup)
- For molecules/atoms that pair a single control with an inline label and need `peer-disabled` styling (text dims + cursor-not-allowed when the control is `disabled`), render a NATIVE `<label>` element with the molecule's own `*LabelVariants` cva — do NOT compose the `Label` atom. The Label atom does not expose a `peer-disabled` axis, and the molecule cannot inject className into it. The native `<label htmlFor={id}>` linked via `React.useId()` (with `id` prop override) provides the click-to-toggle behavior without needing Radix Label's labelable-child de-dup logic (that logic only matters when the label WRAPS the control). Pattern is canonical in `src/atoms/Checkbox/Checkbox.tsx` and `src/molecules/Switch/Switch.tsx`.
- The label wrapper carries `data-slot="<molecule>-root"` and `data-size`, the label element carries `data-slot="<molecule>-label"`, and the control itself sits inside as a `peer` sibling. The label's variants string starts with `'select-none ... peer-disabled:cursor-not-allowed peer-disabled:opacity-50'`. When `label` is omitted, the molecule returns the bare control (no wrapper, no label slot) so `aria-label[ledby]`-only usage stays clean.
- Multi-item radio-style molecules (RadioGroup and any future "list of N controls + label + description per row" wrapper) CANNOT use Tailwind's `peer-disabled:` modifier for the label text — the peer selector only reaches the immediate sibling of the disabled control, and when label + description live in a separate column wrapper (`<span class="flex flex-col">`) they are two DOM levels removed and the modifier silently no-ops. Two viable substitutes: (a) wrap the whole row in `<label htmlFor>` and apply the disabled visual treatment imperatively via `cn(..., item.disabled === true && 'opacity-50 cursor-not-allowed')` based on the descriptor's `disabled` field; (b) set `data-disabled=""` on the row and target it with descendant selectors. Pattern (a) is canonical in `src/molecules/RadioGroup/RadioGroup.tsx`. Bonus: a row-level `<label>` makes the description text clickable too, which feels right for grouped choice UI.
- Multi-item molecules generate per-item DOM ids by deriving from a single `React.useId()` base, e.g. `const itemId = ${baseId}-${item.value};`. The `id` prop on the molecule overrides the generated base. Using the descriptor's `value` as the id suffix guarantees uniqueness within the group and stability across renders — do NOT use the array index, which shifts when items are reordered and breaks label/control pairing mid-render. Same `value` doubles as the React key.

## Discriminated-union molecules (ToggleGroup and similar single/multiple-selection wrappers)
- Radix `ToggleGroup` (and other single/multiple-selection primitives) changes per-item ARIA semantics based on the `type` discriminator: `type='single'` sets `role='radio'` + `aria-checked` on each item (radiogroup semantics) while `type='multiple'` keeps `role='button'` + `aria-pressed`. Tests must use the role matching the mode under test — `getByRole('radio', { name })` for single, `getByRole('button', { name })` for multiple. Counting items across both modes is best done via `container.querySelectorAll('[data-slot=<composed-atom-slot>]')` since the role differs by mode.
- For molecules whose props differ by a discriminator (`type: 'single' | 'multiple'` with different `value` / `defaultValue` / `onValueChange` shapes), define separate `*SingleProps` / `*MultipleProps` interfaces both extending a shared `*BaseProps`, then union them as the public `ToggleGroupProps` type. The forwardRef body narrows via `if (props.type === '<discriminator>')` and destructures inside each branch — pass `type="<discriminator>" as const` explicitly to Radix in each branch so it sees the correct narrowed shape. Spreading the union directly into Radix loses the discriminator and produces "string is not assignable to string[]" errors.
- When composing an atom via Radix's `asChild` on a wrapper item (`<RadixToggleGroup.Item asChild value={...}><Toggle .../></RadixToggleGroup.Item>`), Radix uses its Slot machinery to project state plumbing onto the atom. The atom is the actual DOM `<button>` — its `data-slot` / `data-variant` / `data-size` attributes survive, and so does any explicit `aria-label` passed to it. This is the canonical pattern for variant propagation through Radix selection primitives: pass molecule-level axes to the inner atom directly rather than trying to clone or restyle Radix's own rendered child.

## Data-driven molecules (Table and similar large-N cell layouts)
- For molecules that render N homogeneous cells from an array prop (Table's `headers[]`/`rows[][]`/`footer[]`), prefer styling cells via descendant cva selectors on the root (`[&_th]:px-3 [&_td]:py-3`) over wrapping each cell in a composed atom. Per-cell `<Typography>` wrappers multiply render cost on large data sets; descendant selectors apply the same visual scale axis-by-axis with one cn() call on the root. Reserve atom composition for the SINGLETON slots (Table's `caption` uses Typography variant="caption") where the per-instance cost is one wrapper, not N.
- `react/no-array-index-key` is NOT enabled in the shared eslint config — positional cells/rows can use array index keys directly without `eslint-disable` comments. The default `react/jsx-key` rule only fires when a key is missing entirely.
- Scope test queries to a specific table section with `within(container.querySelector('tbody') as HTMLElement).getByRole('cell', ...)` (or `tfoot`) when the same cell text could appear in multiple sections. The `within()` helper from `@testing-library/react` accepts any HTMLElement, not just the bound render container, and pairs well with `data-slot="table-cell"` / `data-slot="table-head"` queries.

## Portal-based Radix wrapper molecules (Popover, Tooltip, HoverCard, ContextMenu, Select)
- Radix Popover/Tooltip/HoverCard/ContextMenu Content automatically emits `data-state="open|closed"`, `data-side="top|right|bottom|left"`, and `data-align="start|center|end"` AFTER collision detection runs. The molecule's `placement`/`align` axes are the CONSUMER's request — the resolved value may differ if there's no room. Do NOT re-emit `data-placement`/`data-align` from the molecule (it would shadow Radix's collision-aware attrs and mislead consumers reading them in DevTools). Keep `data-size` (a molecule-owned axis) but defer placement/align observability to Radix's own attrs. Tests can rely on `data-side`/`data-align` after `findByRole`.
- When a portal-based molecule conditionally composes a surface atom (Card) AND falls back to inline surface styling for the unwrapped case, the cva must NOT bake the surface classes into the base string — they'd double up with Card's own border/shadow. Pattern: keep `popoverContentVariants` (or equivalent) for sizing + positioning only; export a sibling `*ContentSurfaceVariants` cva for the surface (border + bg + shadow + padding); compose conditionally in `cn(base, !hasCard && surface(...))`. Mirrors the wrapper-frame particle composition pattern from atoms.
- Radix's portaled Content carries `role="dialog"` (Popover) or `role="tooltip"` (Tooltip) etc. by default. The `aria-dialog-name` axe rule fails when a dialog has no accessible name. Satisfy via `contentProps={{ 'aria-label': '...' }}` in tests, or rely on Radix auto-wiring `aria-labelledby` when the consumer provides a title element with a known id (Popover does NOT auto-wire). Document this in molecule READMEs so consumers know to supply one of `title` / `contentProps.aria-label` for accessible popovers.
- `bg-popover` / `text-popover-foreground` (shadcn defaults) are NOT in `src/styles/globals.css` and will silently drop under Tailwind v4. Use `bg-background` / `text-foreground` for portaled surfaces, or add the `--color-popover*` tokens to `@theme` first. Same for `bg-card` / `text-card-foreground` — also missing; Card itself uses them but they silently no-op (a pre-existing issue, not introduced by molecules).
- `contentProps` (the pass-through bag for the Radix Content layer in portal-based molecules) MUST `Omit` `'className'` plus any molecule-owned positioning props (`side`, `align`, `children`). Forgetting `'className'` re-opens the escape hatch through the bag — the same rule that applies to atom per-part bag props (Avatar's `imageProps`, ScrollArea's `viewportProps`/`scrollbarProps`).
- Default to `modal={false}` for Popover-like wrappers (background pointer events stay live; consumer opts in to focus trapping when they truly want dialog semantics). Tooltips have no `modal` axis; HoverCard is implicitly non-modal. Document the default explicitly in the README.
- Radix Tooltip renders its body content TWICE in the DOM when open — once inside the visible `[data-slot=tooltip-content]` div, and again inside a nested visually-hidden `<span role="tooltip">` used as the `aria-describedby` target. `screen.findByText('...')` collides with both copies and throws "Found multiple elements". Scope queries via a `waitFor(() => document.querySelector('[data-slot="tooltip-content"]'))` helper and use `within(visibleContent).getAllByText('...')[0]` to disambiguate. Same gotcha will apply to any other Radix wrapper that emits a visually-hidden a11y twin.
- axe's `region` rule fails on Tooltip a11y tests because Radix portals the Content to `document.body` — outside any `<main>` landmark the test wraps the render in. `role="dialog"` (Popover) is auto-exempted by axe so Popover passes naturally; `role="tooltip"` is NOT exempted. Resolution in component-isolation tests: `axe(baseElement, { rules: { region: { enabled: false } } })`. The consumer's app shell provides the landmark in real usage — the portaled tooltip sitting above page flow is by design.
- Tooltip molecules render their OWN internal `RadixTooltip.Provider` (with the molecule's `delayDuration` default, 300ms) so consumers don't need to wrap their app in a top-level Provider. The Provider's `delayDuration` / `skipDelayDuration` / `disableHoverableContent` are exposed as molecule props for per-instance tuning. Pattern is Tooltip-specific — Popover/HoverCard/ContextMenu don't require a Provider, so they wrap only the Root.
- Radix Select / Listbox primitives call `Element.prototype.scrollIntoView` on the focused item when the listbox opens, and `Element.prototype.hasPointerCapture` / `setPointerCapture` / `releasePointerCapture` during the pointer-driven open path. jsdom implements none of these — the missing-method TypeError fires inside a commit-phase effect, unmounts the test render mid-frame, and downstream queries (e.g. `getByRole('listbox')`) then fail with confusing `root.getElementById is not a function` errors from `dom-accessibility-api`. Polyfill all four as no-ops in `vitest.setup.ts` (same place ResizeObserver lives). Required for any Select/Combobox/Listbox-style portaled menu — Popover/Tooltip/HoverCard/ContextMenu do NOT need these polyfills because they don't scroll-into-view their content.
- For Radix-trigger molecules that own the rendered trigger DOM (Select's wrapper-frame trigger, NativeSelect, future Combobox), expose a `triggerProps?: Omit<RadixTriggerProps, 'className' | 'children'>` pass-through bag for ARIA wiring (`aria-label`, `aria-labelledby`, `id`, focus handlers, `disabled`). Different from the Popover/Tooltip/HoverCard pattern, which receives the trigger as a `React.ReactElement` slot and lets the consumer's element carry its own ARIA — Select's default trigger is internal, so the consumer needs a typed seam to label the combobox. Bag MUST `Omit<..., 'className' | 'children'>` (same className-escape-hatch rule as `contentProps`).
- Discriminated-union `items` slot prop on portaled-listbox molecules (Select, future Combobox): use explicit `type: 'item' | 'separator'` tags and separate exported interfaces (`SelectItemEntry`, `SelectSeparatorEntry`) instead of `type?: 'item'` optionals. Required-discriminator unions narrow correctly in `items.map((item) => item.type === 'separator' ? ... : ...)` without the consumer needing to spell out `as` casts; optional-discriminator unions force `item.type === 'item' || item.type === undefined` checks everywhere. Item-value-as-React-key (`<RadixSelect.Item key={item.value}>`) is canonical; separators have no natural identity so `key={'separator-${index}'}` is acceptable.
- When a Radix wrapper offers BOTH a default styled trigger AND an optional `trigger?: React.ReactElement` override (Select), the override path uses `<RadixSelect.Trigger asChild {...triggerProps}>{trigger}</RadixSelect.Trigger>` — `asChild` projects Radix state plumbing onto the consumer's element, and the molecule's visual axes (`variant`/`size`/`density`/`tone`) become no-ops in this mode (the consumer's element owns the visual treatment). Document this clearly in the README; readers will otherwise expect the axes to still apply. Tests should cover the override path with an assertion that the consumer-supplied element survives (e.g. `expect(trigger.tagName).toBe('BUTTON'); expect(trigger).not.toHaveAttribute('data-slot', 'select-trigger');`).

## Native form-control molecules (NativeSelect and similar wrapper-frame atoms over native widgets)
- Native `<select>` (single, no `multiple`) has implicit ARIA role `combobox` per the modern HTML AAM spec — the same role Radix Select's Trigger emits — so `screen.getByRole('combobox', { name })` works for both NativeSelect and the portaled Select molecule. With `multiple` or `size > 1` the implicit role becomes `listbox`. Native form-control accessible names come from the wrapping `<label htmlFor>`, an `aria-label`, or `aria-labelledby` — the molecule does NOT auto-generate one.
- `<option hidden>` (used as a `disabled hidden value=""` placeholder pattern paired with `defaultValue=""` to render the placeholder selected on mount) is excluded from the ARIA tree, so `screen.getByRole('option', { name, hidden: true })` STILL misses it — testing-library's `hidden: true` option controls whether `aria-hidden` parents are traversed, not the HTML `hidden` attribute on `<option>`. Query directly via DOM (`container.querySelector('option[value=""]')`) and assert `toBeDisabled()` + `toHaveAttribute('hidden')` + `toHaveTextContent(placeholder)`. The visible `<option>` entries remain queryable by `getByRole('option', { name })` as normal.
- Native `<select>` chevron suppression: apply `appearance-none` on the inner `<select>` and render a decorative chevron icon as a sibling inside the wrapper-frame (lucide-react's `ChevronDown` is canonical). The chevron span carries `aria-hidden` and `pointer-events-none` so clicks fall through to the native control and screen readers ignore the duplicate glyph. The OS-default chevron's font-size scales with the form-control's `font-size`, so stripping it gives us pixel-stable alignment across `size: sm | md | lg`.
- `userEvent.selectOptions(select, value)` (from `@testing-library/user-event`) is the canonical way to drive native `<select>` value changes in tests — accepts a string, a number, an HTMLOptionElement, or an array (for `multiple`). Fires `onChange` synchronously with the awaited promise; pair with `expect(select).toHaveValue(...)` to assert the resulting selection. Avoid `fireEvent.change(select, { target: { value } })` — it skips the focus/click sequence that real browsers run and can desync controlled-component tests.
- Data-driven `<select>` content uses a discriminated union mirroring Select's `SelectItemDescriptor` but with the option/group distinction: `{ type: 'option', value, label?, disabled? }` for individual entries and `{ type: 'group', label, options: NativeSelectOptionEntry[] }` for `<optgroup>` clusters. The `type` discriminator is required (not optional) so consumers can't accidentally omit the tag and silently render a malformed entry. The molecule renders `<optgroup>` via `key={'${descriptor.label}-${index}'}` because group labels CAN collide across descriptor arrays (the same fallback pattern Select uses for separators).
- Mutually-exclusive `options` vs `children` API: when both are supplied, `options` wins (the molecule passes `options.map(renderDescriptor)` as the select's child set and ignores `children`). Document the precedence in the README's Do/Don't — silent override is acceptable here because the data-driven path is canonical and the children-fallback is the escape hatch for hand-authored JSX.

## Single-hidden-input molecules with N visual slot cells (InputOTP and similar)
- `input-otp`'s `OTPInput` renders a Fragment containing a `<div data-input-otp-container>` (styled via `containerClassName`) which holds the rendered slot row (children OR `render` prop output) PLUS an absolutely-positioned wrapper containing a single hidden `<input>` (gets all `{...rest}` props including ARIA, data-*, ref). The forwarded ref points at the hidden input. There is no separate "root" attribute the consumer can label, so to expose a `data-slot="input-otp-root"` anchor with variant-resolution data-attrs (`data-length`, `data-size`, `data-invalid`), wrap `OTPInput` in your own outer `<div>` — the one-extra-DOM-node cost is worth the clean separation from the hidden-input's `data-slot="input-otp-control"` and per-slot `data-slot="input-otp-slot"` queries.
- Slots are read out of `OTPInputContext.slots[index]` (shape: `{ char, placeholderChar, isActive, hasFakeCaret }`). A per-slot subcomponent that calls `useContext(OTPInputContext)` is the canonical render approach (matches shadcn's pattern). Pass `size` and `invalid` as parent-resolved props rather than reading them from context — they're molecule-owned axes, not library state. The slot's `isActive` flag is the test-observable hook for caret position; cva's `active: { true, false }` axis (plus `compoundVariants` for `{ invalid: true, active: true } → 'ring-destructive'`) keeps the resolved class names in the snapshot rather than relying on `data-[active=true]:` selector lookups.
- Connected-cell visual style (shared borders between neighbouring cells without doubling up) uses `border-y border-r first:border-l first:rounded-l-md last:rounded-r-md` on each cell — NOT `gap-*` on the parent. This matches the shadcn default and looks like a single rounded group instead of N detached chips. Reserve `gap-*` for OTP variants that intentionally space the cells (e.g. the "phone PIN with dash separator" layout).
- The `input-otp` library exports `REGEXP_ONLY_DIGITS`, `REGEXP_ONLY_CHARS`, and `REGEXP_ONLY_DIGITS_AND_CHARS` as regex-source strings — use these directly when defaulting the `pattern` prop. For numeric OTP the trio is `inputMode="numeric"`, `autoComplete="one-time-code"`, `pattern={REGEXP_ONLY_DIGITS}`. All three must be overridable via props for alphanumeric / mixed-case codes (mobile users get locked out of mixed-case fields when `inputMode="numeric"` forces the digits-only keypad).
- `input-otp`'s `onChange` signature is non-standard: `(newValue: string) => unknown` — NOT a React `ChangeEvent`. The molecule passes it through `{...rest}` unchanged; document this in the README so consumers don't try to read `event.target.value`. `onComplete` fires when the value transitions from `< maxLength` to exactly `maxLength` (one-shot at boundary); useful for auto-submitting verification forms.
- Test the controlled-value rendering by passing `value="12"` directly — input-otp resolves `i = value ?? internalState` so the context slots immediately reflect the prop. Use `fireEvent.change(input, { target: { value: '...' } })` instead of `userEvent.type` for OTP value transitions — the library's internal `selectionchange` listener pairs with native input events fine in jsdom, but userEvent's character-by-character typing path can race with the polyfilled scrollIntoView / setSelectionRange in unstable ways. `getByRole('textbox', { name })` queries the hidden input (it has no explicit role, so it defaults to textbox).
- `containerClassName` on `OTPInput` is the right place to apply Tailwind's `has-[input:disabled]:` modifier — the input is a descendant of the container div, and the `:has()` selector lets the row-level dim style fire without per-cell coordination. Same trick that NativeSelect's wrapper uses for the native `<select>`.
