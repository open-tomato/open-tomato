# Phase 8 — open-tomato progress notes

Append broadly-applicable findings only. Skip task-specific details.

## Findings

- Shared packages live under `packages/shared/*` and are auto-registered via the root `workspaces` glob in `package.json` — no need to manually edit the root for new members.
- Standard devDependency pins in shared packages: `eslint ^9.39.4`, `typescript ^5.9.3`, `vitest ^3.0.0`, `@types/node ^22.13.5`, plus `@open-tomato/eslint-config` and `@open-tomato/typescript-config` at `workspace:^`.
- Convention for pure-TS shared packages (no build step): `main` and `exports["."]` both point to `./src/index.ts`; `type: "module"`; `private: false`; `publishConfig.access: "public"`.
- Standard `scripts` in shared packages: `"lint": "eslint --fix"` and `"test": "vitest run"`. Build scripts only appear when a package ships pre-bundled output (e.g. `logger`).
- Root scripts of note: `bun run preflight`, `bun run publish:dry`, `bun run publish:local`, `bun run changeset:add` (wrapper around the changeset CLI accepting `--level` and `--summary`/`--pkg name:level` flags).
- `globals ^16.0.0` is commonly listed as a devDep alongside `eslint` in shared packages (see `errors`, `logger`) — add it if eslint config requires it; the task spec is authoritative when it doesn't list it.
- Convention for pure-TS shared `tsconfig.json`: `extends @open-tomato/typescript-config/base`, `compilerOptions.noEmit: true`, `include: ["src"]` (or `["src/**/*"]`), `exclude: ["node_modules", "dist"]`. Add `compilerOptions.paths` only when the package exposes subpath exports (see `logger`).
- `bun run changeset:add` only auto-detects packages from *uncommitted* git changes. If you commit before adding the changeset, you must pass `--pkg <name>:<level>` explicitly or the script exits with "no target packages".
- `@open-tomato/eslint-config/base` enables `globals.browser` + `Bun` but leaves `globals.node` commented out. Node-runtime shared packages (errors, logger, types, event-bus, config) re-add `globals.node` via the `globals` devDep; lightweight contract/types packages can get away with a pure `export default baseConfig` re-export, but if any `src/` file touches `process`/`Buffer`/etc., add `globals` to devDeps and spread `globals.node` into `languageOptions.globals`.
- Standard `vitest.config.ts` for Node-runtime shared packages: `import { defineConfig } from 'vitest/config'; export default defineConfig({ test: { environment: 'node' } });`. Use `'jsdom'` only when the package renders DOM (e.g. `cache` provides a React provider).
- Mid-scaffold lint gotcha: `bun lint` reports `import/no-unresolved` on a brand-new package's root config files (e.g. `vitest.config.ts`) when `src/` is still empty. Root cause: the `import/resolver typescript` setting points at `tsconfig.json`, which only includes `src/**/*` — with zero inputs, `tsc` errors with TS18003 and the resolver can't bootstrap. Lint clears as soon as the next stage adds the first `src/` file; don't add a workaround at scaffold time.
- Tsconfig `extends` must reference `@open-tomato/typescript-config/base` (no `.json` suffix). The package's `exports` map only exposes `./base`, `./react`, `./next` — using `@open-tomato/typescript-config/base.json` makes the extends silently fail with TS6053, which cascades into ESLint's typescript resolver and produces misleading `import/no-unresolved` errors on unrelated files. If you see lint complain about resolving e.g. `vitest/config`, check tsconfig extends first.
- Forward type-only imports between sibling `src/` files (e.g. `import type { Foo } from './foo'` where `./foo.ts` will be added in a later stage task) keep ESLint green under `moduleResolution: bundler` but `tsc --noEmit` reports TS2307 until the dependency file lands. Treat the tsc error as expected during multi-stage scaffolds; verify lint passes and move on.
